# Документация и ответы на вопросы к лабораторной работе №4

## 1 Анализ и пример использования контекста с динамическими свойствами на HttpContext.Items
Слово "context" в названии класса хорошо объясняет его логику - мы задаём контекст для HTTP запроса 
и работаем с этим контекстом на протяжении всей цепочки обработки.

	HttpContext.Items - Получает коллекцию «ключ-значение», 
	которую можно использовать для организации и обмена данными между 
	интерфейсами IHttpModule и IHttpHandler во время HTTP-запроса.
	(определение взято с официального сайта Microsoft).

Например, можно хранить и изменять данные о контексте одной веб сессии пользователя, или передавать данные между запросами: 
можно сохранить залогиненного пользователя (или его ID), и при необходимости не обращаться к токену или БД, а к HttpContext.Items["CurrentUserId"].

Без такого словаря пришлось бы передавать/хранить данные используя отдельные объекты или буферы. 
Это работало бы для одного конкретного случая, но как только нужно передавать/хранить разные или похожие данные одновременно - начинались бы проблемы. 
Словарь Items решает эту проблему: всё организованно в одном месте и гибко для масштабирования и разработки в целом.

## 2 FatStruct
Программирование на основе «жирных» структур (часто называемое «Fat Structs») -
это подход к проектированию, ориентированный на данные, 
при котором структуры данных разрабатываются таким образом, чтобы хранить 
все возможные свойства или варианты объекта, вместо использования более мелких, 
специализированных структур с наследованием или объединениями. 
Этот подход направлен на повышение производительности за счет оптимизации использования кэша 
ЦП и минимизации динамического выделения памяти. (взято из AI Summary при гугл запросе)

По сути, если библиотека не знает о своих пользователях и мы хотим использовать FatStruct, нам либо нужно добавить очень очень много полей,
которые покроют все возможные варианты использования библиотеки, либо при желании добавить нового пользователя идти и изменять код и добавлять поля
библиотеки вручную, что протеворечит всем нормам SOLID программирования и простой логики. 

## 5 Реализация задания

### Разделение на библиотеку и потребителя

Библиотека (lab4_properties_lib) содержит базовые настройки для хранения данных: Entity, EntityStore, KeyRegistry, TypedKey.
Потребитель (lab4_properties) содержит всё конкретное — ключи, операции над данными и запуски. 
Библиотека ничего не знает о потребителе, зависимость односторонняя.

Для хранения свойств внутри Entity выбран массив object[].
Обращение к массиву по int индексу быстрее чем поиск по строке в словаре —
у строк дольше имплементация сравнения. int ключ выдаёт реестр, и навигация по данным сводится к простому смещению по индексу.

**TypedKey<T> — дженерик**

Без дженерика ключ работал бы только для одного типа данных. <T> позволяет использовать одну и ту же логику хранения и передачи 
данных для разных типов — так же как контекст позволяет хранить разные данные в одном месте. 

**KeyRegistry как защита от коллизий**
Реестр не позволяет создать два ключа с одинаковым названием или с одинаковым названием и разными типами — это привело бы к 
коллизиям данных.

**IEntityOperation для расширяемости**
Интерфейс живёт в библиотеке, реализации — у потребителя. Это даёт возможность расширять функционал не трогая библиотеку: 
потребитель пишет свои операции, EntityStore вызывает их для каждой сущности.
